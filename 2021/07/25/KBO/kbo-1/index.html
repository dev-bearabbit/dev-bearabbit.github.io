1. AWS Lightsail로 DB 서버 생성하기

Posted on 2021-07-25 In PROJECT , KBO Analytical Dashboard Disqus:

해당 시리즈는 한국프로야구(Korea Baseball Organization) 웹사이트에서 크롤링한 데이터를 분석하여 대시보드를 만드는 과정을 정리합니다.

시작하며

이전 글에서 언급했지만 사실 해당 프로젝트는 개인적으로 진행했었다. 스터디 팀원들과 함께 KBO 홈페이지 데이터 크롤링을 성공한 후, 웹개발을 배워보고 싶은 마음에 해당 데이터로 “내맘대로 만드는 간단 대시보드”를 만들기 시작하였다. 하지만 스터디에서 크롤링한 데이터를 그냥 두는 것은 아깝다는 의견이 많아졌고, 이는 결국 팀원마다 대시보드를 만드는 것으로 결론이 났다. 즉, 다같이 DB 서버 하나를 생성해서 각각의 웹 서버들이 공유하는 방식으로 진행하자는 것이다.

처음 혼자서 웹서버를 올렸던 구조는 AWS의 ec2 프리티어 버전 인스턴스를 하나 생성해서 도커를 설치한 후, 각 컨테이너에 DB, flask를 올리는 방식이다. 물론 DB와 웹서버는 분리해서 사용하는 것이 맞다고 알고 있었지만, 가벼운 사이드 프로젝트라서 금전을 아끼고 싶었다. 그러다가 ec2 무료 사용시간을 모두 소진했고, 마침 스터디 다른 팀원들도 웹 서버를 만든다고 해서 이런 결말을 맞았다.

아키텍처 요약

해당 프로젝트의 구조는 다음과 같다.

  1. 깃허브 액션: KBO 홈페이지 크롤링 및 실시간 데이터 전처리
  2. DB 서버: 데이터 저장 및 대시보드 데이터 전송
  3. 웹 서버: 데이터 분석 및 시각화

참 간단하다. 매일 KBO 공식 홈페이지에 업데이트 되는 자료들을 크롤링한 후 DB에 저장할 수 있는 형태로 전처리하여 DB 서버로 보낸다. 그러면 DB 서버에는 새로운 시합들에 대한 정보가 저장되고, 해당 DB 서버와 연결된 대시보드는 실시간 정보가 반영된 분석 및 시각화를 제공할 수 있다.

하지만 위 구조는 앞으로 만들어질 실시간 데이터만을 처리한다. 즉, 2000년부터 2021년 현재까지 과거 데이터의 경우는 수동적으로 크롤링하여 전처리한 후 DB에 저장해주어야 한다. 이를 위해서 먼저 해야할 일은 사용할 DB 서버 생성, DB 스키마 설계 등 이었다.

DB 서버 생성

DB 서버는 AWS의 lightsail을 사용하기로 결정했다. 그 이유로는 매달 결정되어 있는 가격대와 편리함 등이 꼽혔다. 여러 명이 같이 돈을 지불해야 하는 탓에 금액이 정해져있지 않으면 매달 귀찮은 일이 발생할 것 같았다.
